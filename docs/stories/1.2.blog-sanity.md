## Story

**Como** parceiro não técnico responsável pelo conteúdo da clínica,
**quero** criar, editar, publicar, arquivar e restaurar posts em uma interface editorial simples,
**para que** visitantes encontrem conteúdo do blog pela landing page e pelas rotas públicas sem depender de suporte técnico.

## Acceptance Criteria

1. O projeto usa `next-sanity` e incorpora o Sanity Studio em `/studio/[[...tool]]`, com autenticação e autorização delegadas aos membros e papéis do projeto Sanity; a rota possui metadados `noindex` e nenhum token privado é enviado ao navegador.
2. O Studio apresenta um único tipo editorial principal, “Post”, com os campos mínimos: título, slug único derivado do título, resumo, imagem de capa com texto alternativo, data de publicação, conteúdo rico e estado de arquivamento.
3. Um editor autenticado consegue criar, editar, publicar, despublicar, arquivar e restaurar posts. Arquivamento e restauração são ações nomeadas em português, e posts arquivados permanecem localizáveis em uma visão separada do Studio.
4. A rota pública `/blog` lista somente posts publicados e não arquivados, do mais recente para o mais antigo, com imagem, título, resumo, data e link de leitura; a página possui estado vazio e visual responsivo coerente com a landing page.
5. A rota pública `/blog/[slug]` exibe somente um post publicado e não arquivado, com metadados de SEO, capa, data e Portable Text. Slug inexistente, não publicado ou arquivado resulta em `404` sem vazamento de dados.
6. A landing page exibe uma seção de prévia do blog com até três posts publicados e não arquivados e um CTA para `/blog`. A navegação principal passa a exibir o link “Blog” apenas quando a rota estiver funcional.
7. Toda entrada dinâmica em GROQ usa parâmetros (`$slug`, limites numéricos controlados) em vez de interpolação. Não existe SQL nessa integração. As projeções retornam apenas os campos necessários.
8. Conteúdo rico não usa `dangerouslySetInnerHTML`. Links vindos do CMS aceitam somente caminhos internos e protocolos `http:`/`https:`; links externos usam `rel="noopener noreferrer"`. Imagens exigem texto alternativo editorial.
9. Falhas temporárias do Sanity não derrubam a landing page nem a listagem: a prévia pode ser omitida e `/blog` mostra estado indisponível/vazio controlado, sem expor detalhes internos.
10. A implementação mantém Server Components como padrão, separa contrato de domínio, repositório Sanity e componentes de apresentação, sem estado global ou API CRUD paralela.
11. O desenvolvimento segue TDD: testes falham antes da implementação correspondente e cobrem mapeamento/validação de dados, consultas parametrizadas, filtragem de arquivados, URLs seguras, renderização da listagem, detalhe e prévia da landing.
12. `npm run lint`, `npm run typecheck`, `npm test`, `npm run build` e `npm run test:e2e` passam; as rotas `/`, `/blog`, `/blog/[slug]` e `/studio` não apresentam erro de console nos cenários testáveis.

## Dev Notes

### Previous Story Insights

- A aplicação Next.js real está em `src/web`; a raiz apenas delega scripts. A landing já usa conteúdo tipado e provider assíncrono, permitindo adicionar um adaptador sem acoplar a UI ao CMS. [Source: `docs/framework/source-tree.md#estrutura-de-código`] [Source: `docs/architecture/content-architecture.md#contrato-independente-de-cms`]
- Fontes locais e placeholders mantêm o build independente de recursos remotos; qualquer falha de conteúdo deve degradar sem quebrar a landing. [Source: `docs/stories/1.1.landing-page-institucional.md#completion-notes-list`]

### Architecture and Boundaries

- A direção aprovada é `next-sanity`, schemas em `src/web/sanity/schemaTypes`, queries em `src/web/sanity/queries`, adapter em `src/web/content/providers/sanity` e Studio incorporado em `/studio/[[...tool]]`. [Source: `docs/architecture/content-architecture.md#sanity-futuro`]
- Componentes de apresentação não podem importar `next-sanity`, GROQ ou tipos do Sanity; a dependência permitida é página → provider → contrato. [Source: `docs/architecture/content-architecture.md#regra-de-dependência`]
- Rotas previstas são `/blog`, `/blog/[slug]` e `/studio/[[...tool]]`; Draft Mode e Visual Editing são opcionais e não são necessários para este escopo. [Source: `docs/architecture/content-architecture.md#blog-futuro`]
- Server Components são o padrão e `"use client"` deve ficar na menor fronteira interativa. [Source: `docs/architecture/frontend-architecture.md#fronteira-serverclient`]

### Security and Configuration

- Tokens privados devem ficar no servidor e nunca usar prefixo `NEXT_PUBLIC_`; conteúdo publicado em dataset público não requer token. [Source: `docs/architecture/content-architecture.md#sanity-futuro`]
- O código não pode inserir conteúdo com `dangerouslySetInnerHTML`; links externos devem ser construídos por funções dedicadas e usar `rel="noopener noreferrer"`. [Source: `docs/framework/coding-standards.md#segurança`]
- Development, Preview e Production usam variáveis separadas; origens de preview/produção devem ser cadastradas nos CORS Origins do Sanity. [Source: `docs/architecture/deployment.md#etapa-sanity`]

### Project Structure

- Novas rotas ficam em `src/web/app/blog` e `src/web/app/studio`; schemas, queries e adapter seguem a árvore já reservada para a etapa Sanity. [Source: `docs/framework/source-tree.md#etapa-sanity`]
- Tipos de conteúdo ficam em `src/web/types`, seções recebem dados prontos e não buscam conteúdo internamente. [Source: `docs/framework/coding-standards.md#typescript`] [Source: `docs/framework/coding-standards.md#react-e-nextjs`]

### Testing

- Usar Vitest + Testing Library para funções puras, componentes e acessibilidade observável; não testar classes Tailwind nem detalhes internos. [Source: `docs/framework/testing-strategy.md#vitest-e-testing-library`]
- Usar Playwright para navegação, teclado, ausência de overflow e erros de console nos viewports definidos. [Source: `docs/framework/testing-strategy.md#playwright`]
- Todos os gates são obrigatórios: lint, typecheck, testes, build e E2E. [Source: `docs/framework/testing-strategy.md#quality-gates`]
